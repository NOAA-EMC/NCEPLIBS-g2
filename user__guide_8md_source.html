3 # Introduction
4 
5 This document briefly describes the routines available for encoding/decoding
6 GRIB Edition 2 (GRIB2) messages. A basic familiarity with GRIB is assumed.
7 
8 A GRIB Edition 2 message is a machine independent format for storing
9 one or more gridded data fields. Each GRIB2 message consists of the
10 following sections:
11 
12 <pre>
13 SECTION 0 - Indicator Section
14 SECTION 1 - Identification Section
15 SECTION 2 - (Local Use Section) - optional }
16 SECTION 3 - Grid Definition Section } }
17 SECTION 4 - Product Definition Section } } }(repeated)
18 SECTION 5 - Data Representation Section } }(repeated) }
19 SECTION 6 - Bit-map Section }(repeated) } }
20 SECTION 7 - Data Section } } }
21 SECTION 8 - End Section } } }
22 </pre>
23 
24 Sequences of GRIB sections 2 to 7, 3 to 7, or sections 4 to 7 may be repeated
25 within a single GRIB message. All sections within such repeated sequences
26 must be present and shall appear in the numerical order noted above.
27 Unrepeated sections remain in effect until redefined.
28 
29 The above overview was taken from WMO's FM 92-XII GRIB description
30 of the experimental GRIB Edition 2 form.
31 
32 # GRIB2 Encoding Routines
33 
34 Since a GRIB2 message can contain gridded fields for many parameters on
35 a number of different grids, several routines are used to encode a message.
36 This should give users more flexibility in how to organize data
37 within one or more GRIB2 messages.
38 
39 To start a new GRIB2 message, call subroutine gribcreate(). It
40 codes Sections 0 and 1 at the beginning of the message. This routine
41 must be used to create each message.
42 
43 Subroutine addlocal() can be used to add a Local Use Section (Section 2).
44 Note that section is optional and need not appear in a GRIB2 message.
45 
46 Subroutine addgrid() is used to encode a grid definition into Section 3.
47 This grid definition defines the geometry of the the data values in the
48 fields that follow it. ADDGRID() can be called again to change the grid
49 definition describing subsequent data fields.
50 
51 Each data field is added to the GRIB2 message using routine ADDFIELD(),
52 which adds Sections 4, 5, 6, and 7 to the message.
53 
54 After all desired data fields have been added to the GRIB2 message, a
55 call to routine GRIBEND() is needed to add the final section 8 to the
56 message and to update the length of the message. A call to GRIBEND()
57 is required for each GRIB2 message.
58 
59 # GRIB2 Decoding Routines
60 
61 Subroutine GB_INFO() can be used to find out how many Local Use sections
62 and data fields are contained in a given GRIB2 message. In addition,
63 this routine also returns the number of octets of the largest Local Use
64 section in the message. This value can be used to ensure that the
65 output array of subroutine GETLOCAL() (described below) is dimensioned
66 large enough.
67 
68 Subroutine GETLOCAL() will return the requested occurrence of Section 2
69 from a given GRIB2 message.
70 
71 GF_GETFLD can be used to get all information pertaining to the nth
72 data field in the message. The subroutine returns all the unpacked values
73 for each Section and Template in a Fortran 90 derived type gribfield,
74 which is defined in module GRIB_MOD. An option exists that lets the
75 user decide if the subroutine should unpack the Bit-map (if
76 applicable) and the data values or just return the field description
77 information.
78 Note that derived type gribfield contains pointers to dynamically
79 allocated space that holds the contents of all arrays, and users are encouraged
80 to free up this memory, when it is no longer needed, by an explicit call
81 to subroutine GF_FREE().
82 
83 # Extracting GRIB2 Fields from a GRIB2 file
84 
85 Subroutine GETGB2() can be used to extract a specified field from a file
86 containing many GRIB2 messages. GETGB2() searches an index to find the
87 location of the user specified field. The index can be supplied from a
88 seperate GRIB2 index file, or it can be generated internally.
89 
90 The GRIB2 file (and the index file, if supplied) must be opened with
91 a call to subroutine BAOPEN() prior to the call to GETGB2.
92 
93 The decoded information for the selected GRIB field is returned in a
94 derived type variable, gfld. Gfld is of type gribfield, which is defined
95 in module grib_mod, so users of this routine will need to include
96 the line "USE GRIB_MOD" in their calling routine. Each component of the
97 gribfield type is described in the OUTPUT ARGUMENT LIST in the docblock
98 for subroutine GETGB2() below.
99 
100 Note that derived type gribfield contains pointers to many arrays of data.
101 The memory for these arrays is allocated when the values in the arrays
102 are set, to help minimize problems with array overloading. Because of this,
103 users are encouraged to free up this memory, when it is no longer
104 needed, by an explicit call to subroutine GF_FREE().
105 
106 Example usage:
107 
108 @code
109  use grib_mod
110  type(gribfield) :: gfld
111  integer,dimension(200) :: jids,jpdt,jgdt
112  logical :: unpack=.true.
113  ifile=10
114  ! Open GRIB2 file
115  call baopenr(ifile,"filename",iret)
116  .
117  ! Set GRIB2 field identification values to search for
118  jdisc=
119  jids(?)=
120  jpdtn=
121  jpdt(?)=
122  jgdtn=
123  jgdt(?)=
124 
125  ! Get field from file
126  call getgb2(ifile,0,j,jdisc,jids,jpdtn,jpdt,jgdtn,jgdt,
127  & unpack,j,gfld,iret)
128 
129  ! Process field ...
130  firstval=gfld%fld(1)
131  lastval=gfld%fld(gfld%ndpts)
132  fldmax=maxval(gfld%fld)
133  fldmin=minval(gfld%fld)
134 
135  ! Free memory when done with field
136  call gf_free(gfld)
137 
138  stop
139  end
140 @endcode
